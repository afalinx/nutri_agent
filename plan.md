# План разработки NutriAgent

> Общий срок: **4 недели** (MVP) + 1 неделя (полировка).
> Подход: от ядра к интерфейсу. Сначала AI работает в консоли, потом — в API, потом — на экране.

---

## Спринт 1: Фундамент (дни 1–5)

**Цель:** Инфраструктура поднята, БД работает, CRUD-операции доступны через Swagger.

| # | Задача | Результат | Критерий готовности |
|---|--------|-----------|---------------------|
| 1.1 | Инициализировать backend (FastAPI + Poetry/requirements) | Проект запускается, `/docs` отвечает | `curl localhost:8000/docs` → 200 |
| 1.2 | Написать `docker-compose.yml` | PostgreSQL (pgvector) + Redis стартуют | `docker-compose up` без ошибок, `psql` подключается |
| 1.3 | Создать `.env.example` | Все переменные задокументированы | DATABASE_URL, REDIS_URL, OPENROUTER_API_KEY, LLM_MODEL_NAME, SECRET_KEY |
| 1.4 | Описать модели SQLAlchemy | User, Recipe, MealPlan | Модели создаются в БД через Alembic |
| 1.5 | Настроить Alembic | Миграции работают | `alembic upgrade head` применяет схему |
| 1.6 | CRUD API: профиль пользователя | POST/GET/PUT `/api/users` | Swagger: создать юзера → получить → обновить |
| 1.7 | Расчёт целевого калоража | Формула Миффлина-Сан Жеора в `core/skills/calculator.py` | Unit-тест: муж, 25 лет, 80 кг, 180 см, moderate → 2500 ±50 ккал |
| 1.8 | Настроить логирование (loguru) | INFO/DEBUG/ERROR уровни | Логи пишутся в stdout с уровнем и таймстемпом |

**Блокер для спринта 2:** задачи 1.1, 1.2, 1.4.

## Спринт 2: AI-ядро и RAG (дни 6–14)

**Цель:** В консоли можно запустить скрипт, который генерирует валидный JSON рациона на 1 день.

| # | Задача | Результат | Критерий готовности |
|---|--------|-----------|---------------------|
| 2.1 | Собрать датасет рецептов (50–100 шт) | JSON-файл с рецептами (title, ingredients, КБЖУ, tags) | Минимум 50 рецептов, покрыты категории: завтрак/обед/ужин/перекус |
| 2.2 | Скрипт загрузки рецептов + эмбеддинги | `scripts/seed_recipes.py` | Рецепты в таблице `recipes`, поле `embedding` заполнено |
| 2.3 | RAG Retriever | `core/rag/retriever.py` — поиск топ-K рецептов | Тест: запрос «завтрак без глютена» → результаты не содержат глютен |
| 2.4 | Фильтрация аллергенов | SQL-фильтр до векторного поиска | Тест: юзер с аллергией «орехи» → ни один рецепт с орехами не возвращается |
| 2.5 | Системный промпт | `core/agent/prompts/meal_plan.yml` | Промпт вынесен из кода, переменные подставляются через Jinja2 |
| 2.6 | Pydantic-схемы для Structured Output | `core/agent/schemas.py` | JSON Schema для дневного плана, LLM отвечает валидным JSON |
| 2.7 | Calculator Skill — валидация | `core/skills/calculator.py` — пересчёт калорий | Тест: JSON с завышенными калориями → ошибка `ValidationError` |
| 2.8 | Оркестратор (без рефлексии) | `core/agent/orchestrator.py` | Скрипт: профиль → RAG → LLM → JSON рациона на 1 день |
| 2.9 | Цикл рефлексии | Retry при ошибке валидации (макс. 3 итерации) | Тест: намеренно некорректный ответ → 2-я попытка → корректный JSON |
| 2.10 | Integration-тест RAG + Agent | Полный пайплайн в консоли | `python -m core.agent.orchestrator` → валидный план, отклонение ≤ 5% |

**Блокер для спринта 3:** задачи 2.8, 2.9 (рабочий оркестратор).

## Спринт 3: Асинхронность и API (дни 15–21)

**Цель:** Фронтенд может запустить генерацию через API и получить результат по `task_id`.

| # | Задача | Результат | Критерий готовности |
|---|--------|-----------|---------------------|
| 3.1 | Celery worker | `worker/tasks.py` — задача `generate_meal_plan` | Worker стартует в Docker, принимает задачи |
| 3.2 | `POST /api/generate-plan` | Принимает `user_id` + параметры, возвращает `task_id` | Swagger: отправить запрос → получить `{ task_id: "..." }` |
| 3.3 | `GET /api/tasks/:task_id` | Возвращает `{ status, result }` | Статусы: PENDING → GENERATING → READY/FAILED |
| 3.4 | Сохранение результата в MealPlan | Celery task пишет `plan_data` в БД | После READY: `GET /api/plans/:id` → полный JSON плана |
| 3.5 | Агрегация списка покупок | `core/skills/aggregator.py` | `GET /api/plans/:id/shopping-list` → список без дубликатов, суммированные граммовки |
| 3.6 | Генерация на 7 дней | Оркестратор вызывается 7 раз (по дню) | Полный недельный план в `plan_data`, все 7 дней |
| 3.7 | Обработка ошибок | Retry + fallback при таймаутах LLM | Тест: имитация таймаута → статус FAILED с логом, сервер не падает |
| 3.8 | E2E-тест | FastAPI TestClient | Тест: POST /generate-plan → poll GET /tasks/:id → READY → plan_data валиден |

**Блокер для спринта 4:** задачи 3.2, 3.3 (API-контракт для фронтенда).

**Цель:** Рабочий веб-интерфейс — от опросника до списка покупок.

| # | Задача | Результат | Критерий готовности |

## Definition of Done (общий для всех задач)

Задача считается выполненной, когда:

1. **Код работает** — функционал можно проверить руками (Swagger / браузер / консоль)
2. **Критические пути покрыты тестами** — unit-тест на Calculator, integration-тест на RAG, e2e-тест на API-пайплайн
3. **Нет критических ошибок** — логи уровня ERROR не появляются при нормальном сценарии
4. **Код в Git** — коммит с осмысленным сообщением

## Риски и контрмеры

| Риск | Вероятность | Контрмера | Ответственный момент |
|------|-------------|-----------|----------------------|
| LLM возвращает невалидный JSON | Средняя | Pydantic-валидация + retry до 3 раз | Спринт 2, задача 2.6 |
| Генерация > 60 секунд | Средняя | Генерировать по 1 дню, параллельно через Celery | Спринт 3, задача 3.6 |
| Мало рецептов = однообразные планы | Высокая | Приоритет на разнообразие категорий (50 рецептов минимум) | Спринт 2, задача 2.1 |
| Рефлексия не сходится за 3 итерации | Низкая | Отдавать лучший вариант + статус `PARTIALLY_VALID` | Спринт 2, задача 2.9 |
| Фронтенд не успевается | Средняя | MVP без графиков и авторизации, только визард + карточки | Спринт 4 |
| OpenRouter API упал / лимиты | Низкая | Retry с exponential backoff, fallback-модель в конфиге | Спринт 3, задача 3.7 |
