# AI-ТЗ: Интеллектуальная система генерации персонализированных планов питания

**Статус проекта:**  ready for development
**Тип архитектуры:** Модульный монолит (Modular Monolith)  

---

## 1. Цели и сценарии (Goals & Scenarios)
**Зачем мы это делаем:** 
Разработать систему для генерации персонализированных рационов питания, которая решает главную проблему LLM в медицине/диетологии — "галлюцинации". Система должна опираться на верифицированные данные (RAG) и жесткую математическую проверку (валидаторы).

**Основные сценарии:**
1. Пользователь заполняет онбординг-опросник (цели, физиология, аллергии).
2. Система асинхронно генерирует план питания на неделю с учетом КБЖУ.
3. Пользователь просматривает план, рецепты и получает агрегированный список покупок.

## 2. Контекст проекта (Project Context)
Проект разрабатывается как выпускная квалификационная работа (диплом). Главная научная и инженерная ценность заключается не в красивом интерфейсе, а во внедрении пайплайна **"Генерация (LLM) -> Верификация (Справочник/Скиллы) -> Корректировка"**. 
Система состоит из единого Backend-приложения, которое оркестрирует запросы к LLM, работает с векторной БД и отдает данные Frontend-клиенту по REST API.

## 3. Метрики успеха (Success Metrics)

Для оценки успешности проекта используются метрики на трех уровнях архитектуры:

### 3.1. Метрики качества AI и RAG (AI & Retrieval)
* **Точность поиска (RAG Relevance):** Векторная БД (`pgvector`) стабильно возвращает релевантные рецепты, строго исключая блюда с аллергенами, указанными пользователем (достигается за счет гибридного поиска: вектор + жесткие SQL-фильтры).
* **Отсутствие галлюцинаций (Groundedness):** LLM формирует рацион **только** на основе контекста, переданного из векторной БД, не выдумывая несуществующих продуктов или ложных показателей КБЖУ.
* **Стабильность рефлексии (Validation Stability):** Пайплайн валидации (LLM -> Калькулятор бэкенда -> LLM) успешно сходится к математически корректному результату не более чем за 3 итерации (цикла рефлексии).

### 3.2. Системные и инфраструктурные метрики (System & Backend)
* **Стабильность API и Оркестрации:** Асинхронный пайплайн (`FastAPI` + `Celery`) корректно отрабатывает 100% запросов на генерацию, не блокируя основной поток сервера и успешно обрабатывая таймауты или ошибки от стороннего LLM-провайдера (настроены retry-механизмы).
* **Консистентность форматов (Structured Output):** LLM стабильно формирует и отдает данные в строгом формате JSON (схемы Pydantic), что позволяет бэкенду без ошибок парсить ответ в БД на каждом запросе.
* **Производительность:** Время векторного поиска по базе рецептов занимает менее 200 мс, а полный цикл асинхронной генерации дневного рациона не превышает 30-40 секунд.

### 3.3. Продуктовые метрики (Product/User)
* **Точность попадания в цель:** Итоговый сгенерированный рацион отклоняется от заданного пользователем целевого калоража (КБЖУ) не более чем на ±5%.
* **Полнота результата:** Система способна бесперебойно сформировать связный план питания на заданный период (например, 7 дней) и выдать корректно сагрегированный список покупок без дубликатов ингредиентов.

## 4. Технические требования (Tech Stack)
* **Frontend:** Next.js (React), Tailwind CSS, Shadcn/UI.
* **Backend:** Python 3.11+, FastAPI (Модульный монолит).
* **AI & Orchestration:** PydanticAI (обеспечивает типизацию и автоматическую рефлексию при валидации) с использованием ручек OpenRouter для запросов.
* **База данных:** PostgreSQL + `pgvector` (хранение юзеров, рационов и векторных эмбеддингов рецептов).
* **Асинхронные задачи:** Celery + Redis (для долгих запросов к LLM и работы агента в фоне).
* **Инфраструктура и конфигурация:** Docker, Docker Compose. Управление секретами через `pydantic-settings` (чтение из `.env`). Все секреты и настройки среды передаются строго через переменные окружения; в корне проекта — файл `.env.example` с описанием ключей (без значений): `DATABASE_URL`, `REDIS_URL`, `OPENROUTER_API_KEY`, `LLM_MODEL_NAME`.
* **Наблюдаемость (Observability):** Библиотека `loguru` (или настроенный `logging`) для структурированных логов: уровень **INFO** — старт/стоп генерации, успешные HTTP-запросы; **DEBUG** — сырые ответы от LLM (для отладки галлюцинаций), скоры релевантности RAG; **ERROR** — таймауты API, ошибки парсинга Pydantic, падения воркеров Celery.
* **Документация API:** Автогенерируемая OpenAPI/Swagger документация FastAPI доступна по адресу `/docs`.

## 5. Архитектура и данные (Architecture & Data)

Так как используется **Modular Monolith**, весь бэкенд живет в одном репозитории, но разделен на логические слои (модули):
* `api/` — FastAPI роуты (контроллеры) и автодокументация Swagger.
* `core/agent/` — Логика агента, вызов LLM, пайплайн генерации.
* `core/agent/prompts/` — Директория с шаблонами системных промптов в формате `.yml` или `.md` (изоляция текстов от Python-кода; промпты не хардкодятся в коде).
* `core/skills/` — Детерминированные модули-инструменты (Calculator, Validator).
* `core/rag/` — Логика векторизации и поиска по `pgvector`.
* `db/` — Модели SQLAlchemy, миграции Alembic.
* `worker/` — Celery таски для вызова агента в фоне.

### 5.1. Модель данных (Схема БД)

База данных проектируется в реляционной парадигме (PostgreSQL) с использованием расширения `pgvector` и типа данных `JSONB` для гибкого хранения сгенерированных структур.

**1. Таблица `users` (Профиль пользователя)**
* Хранит авторизационные и физиологические данные, необходимые для расчета КБЖУ.
* `id` (UUID, Primary Key)
* `email` (String, Unique)
* `password_hash` (String)
* `age`, `weight_kg`, `height_cm` (Integer)
* `gender`, `activity_level`, `goal` (Enum) — пол, уровень активности, цель.
* `allergies` (Array[String] / JSONB) — список аллергенов для жесткой фильтрации.
* `target_calories` (Integer) — вычисленная норма калорий.
* `created_at` (Timestamp)

**2. Таблица `recipes` (Справочник рецептов)**
* База эталонных блюд для RAG-системы.
* `id` (UUID, Primary Key)
* `title` (String)
* `description` (Text) — пошаговый рецепт.
* `ingredients` (JSONB) — массив ингредиентов с граммовками.
* `calories`, `protein`, `fat`, `carbs` (Float) — нутриенты на 1 порцию.
* `embedding` (Vector) — векторное представление рецепта для семантического поиска.
* `tags` (Array[String]) — теги (веган, без глютена, завтрак).

**3. Таблица `meal_plans` (Сгенерированные рационы)**
* Хранит результат работы AI-агента и привязывается к пользователю.
* `id` (UUID, Primary Key)
* `user_id` (UUID, Foreign Key -> `users.id`)
* `status` (Enum: `PENDING`, `GENERATING`, `READY`, `FAILED`) — статус фоновой генерации в Celery.
* `start_date`, `end_date` (Date) — период, на который составлен план.
* `plan_data` (JSONB) — строго структурированный JSON от LLM (см. схему ниже).
* `created_at` (Timestamp)

### 5.2. Контракт данных (JSON Schema для plan_data)
Чтобы обеспечить стабильную работу Frontend-дашборда, LLM обязана (через Structured Outputs) возвращать данные в следующем формате. Этот же формат парсится в поле `plan_data`:

```json
{
  "total_days": 7,
  "daily_target_calories": 2000,
  "days": [
    {
      "day_number": 1,
      "date": "2026-03-05",
      "total_calories": 1950,
      "meals": [
        {
          "type": "breakfast",
          "recipe_id": "uuid-из-базы-рецептов",
          "title": "Овсяная каша с ягодами",
          "calories": 450,
          "ingredients_summary": [
            {"name": "Овсянка", "amount": 60, "unit": "g"}
          ]
        },
        {
          "type": "lunch",
          "recipe_id": "uuid-из-базы-рецептов",
          "title": "Куриная грудка с рисом",
          "calories": 600,
          "ingredients_summary": [
            {"name": "Куриное филе", "amount": 150, "unit": "g"}
          ]
        }
      ]
    }
  ]
}
```

*(Примечание: `recipe_id` гарантирует, что блюдо взято из базы RAG; полный текст рецепта фронтенд подтянет по этому ID из таблицы `recipes`.)*

*Черновик разметки данных, возможно при разработке поменяется.*

### 5.3. Агентская архитектура и пайплайн рефлексии (Agent Core)
Для минимизации галлюцинаций и обеспечения жестких рамок работы ИИ применяется архитектура «Агент с инструментами» (Tool-calling) и паттерн **рефлексии (Actor–Critic)**.

**Best practices в ядре:**
1. **Разделение логики и промптов:** Промпты хранятся в `.yml`/`.md` в `core/agent/prompts/`. Python только подставляет переменные (контекст пользователя, найденные рецепты) через шаблонизатор (например, Jinja2).
2. **Нулевая температура:** LLM вызывается с `temperature=0` для максимальной детерминированности вывода.
3. **Structured Outputs:** Общение с LLM строго через JSON Schema (OpenAI/OpenRouter), без «просто текста».

**Жизненный цикл генерации (акты рефлексии):**
1. **Сбор контекста (Retrieval):** Бэкенд забирает профиль пользователя и делает векторный поиск (RAG) по базе `recipes`, формируя пул из ~20 разрешенных блюд.
2. **Генерация черновика (Actor):** Агент получает промпт и контекст, собирает план на день и выдает черновой JSON.
3. **Валидация (Critic):** В работу вступают Skills (Python): `CalculatorSkill` суммирует калории ингредиентов из JSON-ответа.
4. **Акт рефлексии (Self-Correction):** Если калькулятор выявляет ошибку (например, сумма калорий 2500 вместо 2000), агент получает программную ошибку: `Validation Error: Target calories exceeded by 500`.
5. **Исправление:** LLM делает повторный вызов (retry) с учетом этой ошибки, заменяя блюда. Максимум итераций рефлексии ограничен (например, 3 попытки), после чего возвращается лучший вариант или статус `FAILED` с логированием.
